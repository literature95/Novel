## 二、严重问题（高优先级 🔴）

### 2.1 API Key 明文存储在 localStorage

**文件**：[store.jsx#L697-L699](file:///d:/Documents/Desktop/Novel/src/store.jsx#L697-L699) + project.js

AI API Key 通过 settings 的 `apiKey` 字段直接存储在 `localStorage` 的 `snowflake-novel-project` 键中，**任何能访问该域名 localStorage 的脚本都可以读取**。

**风险等级**：高危 —— 用户 OpenAI/Claude API Key 泄露即意味着经济损失。

**修复建议**：
- 短期：API Key 只在内存中保持（sessionStorage + 不持久化）
- 中期：通过 Next.js API Route 做后端代理，API Key 仅存服务端环境变量
- 提示用户：在 Settings 页面添加 "API Key 仅存储在本地浏览器" 的安全提示

## 三、架构问题（高优先级 🔴）

### 3.1 store.jsx 单体膨胀 —— 已达 1200+ 行

**文件**：[store.jsx](file:///d:/Documents/Desktop/Novel/src/store.jsx)（1246 行）

`projectReducer` 包含约 **50+ 个 case 分支**，处理角色、地点、魔法、时间线、约束、伏笔、章节、场景、三幕、五线、协同、分册等所有领域的 CRUD。

`NovelProvider` 组件中声明了 **20+ 个 useState**：

```js
const [view, setView] = useState('home')
const [phase, setPhase] = useState(0)
const [chapterIdx, setChapterIdx] = useState(0)
const [sceneIdx, setSceneIdx] = useState(0)
const [generating, setGenerating] = useState(false)
const [saveStatus, setSaveStatus] = useState('saved')
const [editingCharIdx, setEditingCharIdx] = useState(0)
const [editingRuleIdx, setEditingRuleIdx] = useState(0)
const [editingLocIdx, setEditingLocIdx] = useState(0)
const [editingMagicIdx, setEditingMagicIdx] = useState(0)
const [editingTimelineIdx, setEditingTimelineIdx] = useState(0)
const [editingFsIdx, setEditingFsIdx] = useState(0)
const [editingVolumeIdx, setEditingVolumeIdx] = useState(0)
const [editingChapterIdx, setEditingChapterIdx] = useState(0)
const [selectedNodeKey, setSelectedNodeKey] = useState(null)
const [nodeWorkflow, setNodeWorkflow] = useState(null)
const [workflowOpen, setWorkflowOpen] = useState(false)
// ... 还有更多
```

**问题**：
1. 任何一个 state 变化都会导致 Provider 重新渲染，进而触发所有消费者的渲染
2. 每个 editing 索引的 add/update/delete 逻辑高度重复（角色、地点、魔法、时间线、伏笔、分册、章节等各有一套几乎相同的 CRUD 模式）
3. 添加新领域需要同时修改 reducer + Provider 中的多个 callback，违反开闭原则

**修复建议**：

```
方案 A（推荐）：创建通用实体管理 Hook
┌──────────────────────────────────────────────────┐
│  useEntityList(projectKey, dispatchTypes)         │
│  → 返回 { items, add, update, delete, editIdx }  │
│  → 替代所有 editingXxxIdx useState                │
└──────────────────────────────────────────────────┘

方案 B：将 Reducer 按领域拆分
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ character    │  │ chapter      │  │ foreshadow   │
│ reducer      │  │ reducer      │  │ reducer      │
└──────┬───────┘  └──────┬───────┘  └──────┬───────┘
       └─────────────────┴─────────────────┘
                    combineReducers()
```

### 3.2 每次更新触发全量 `enrichProject()` —— 性能热点

**文件**：[store.jsx#L56-L60](file:///d:/Documents/Desktop/Novel/src/store.jsx#L56-L60)

```js
function finalize(state) {
  const next = enrichProject(state)
  return isValidProject(next) ? next : state
}
```

`enrichProject` 被嵌套调用在几乎每一个 reducer 分支中（`return finalize({...state, ...})`），这意味着**每次角色名称修改、每添加一条时间线、每编辑一个场景节拍**，都会触发：全量节点状态重算 + 进度聚合 + 依赖检查。

**影响**：对于 60 章 × 4 场景 = 240 场景的项目，频繁编辑时性能退化明显。

**修复建议**：
- 使用 `useMemo` + 依赖追踪，仅在相应数据变更时才重新计算
- 引入 `dirty flag` 机制：在 reducer 中只标记脏数据，在 `useMemo` 中批量重算
- `enrichProject` 内部增加增量计算路径

## 四、性能问题（中优先级 🟡）

### 4.1 无任何代码分割 (Code Splitting)

**文件**：[App.jsx](file:///d:/Documents/Desktop/Novel/src/App.jsx#L1-L42)

App.jsx 在顶层静态 import 了全部 4 个主视图 + 15+ 个结构面板组件。但用户实际访问时通常只使用 1 个视图。

**当前 bundle 结构**（所有代码打包到一个 chunk）：

```
bundle.js (~400KB+)
├── HomeView, StructureView, ContentView, SettingsView  ← 同时加载
├── 所有 15 个结构面板组件                                ← 同时加载
├── 所有 11 个 lib 引擎模块                               ← 同时加载
└── React + ReactDOM + Lucide                            ← 框架
```

**修复建议**：
```jsx
// App.jsx - 懒加载视图
const HomeView = lazy(() => import('./components/home/HomeView'))
const StructureView = lazy(() => import('./components/structure/StructureView'))
const ContentView = lazy(() => import('./components/content/ContentView'))
const SettingsView = lazy(() => import('./components/settings/SettingsView'))

// 外层包裹 Suspense
<Suspense fallback={<LoadingSpinner />}>
  {view === 'home' && <HomeView />}
  {view === 'structure' && <StructureView />}
  ...
</Suspense>
```

预期效果：首屏 bundle 减少约 60-70%。

### 4.2 无 React.memo / useMemo 优化

所有组件都没有使用 `React.memo` 包裹，包括频繁渲染列表的场景项、角色卡片等。每当 Provider 中任意 state 变化时，所有后代组件都会重新渲染。

**修复建议**：
- `Card`、`Badge`、`ProgressBar` 等纯展示组件使用 `React.memo` 包裹
- 列表渲染中的每个项使用 `React.memo` + key 优化
- Provider 中的复杂计算（`healthScore`、`globalScan` 等）已使用 `useMemo`，这点做得不错

### 4.4 localStorage 每次更新都保存

**文件**：[store.jsx#L687-L693](file:///d:/Documents/Desktop/Novel/src/store.jsx#L687-L693)

```js
useEffect(() => {
  setSaveStatus('saving')
  clearTimeout(saveTimerRef.current)
  saveTimerRef.current = setTimeout(() => {
    const ok = saveProject(project)
    setSaveStatus(ok ? 'saved' : 'error')
  }, 400)
}, [project])
```

400ms 防抖做得不错，但 `project` 是一个大对象，作为 `useEffect` 的依赖会导致深度比较。实际上任何字段变更都会触发这里的逻辑，包括在快速连续编辑（如打字）时。

**修复建议**：
- 使用版本号（`project._version`）替代全量依赖比较
- 或者使用 `useRef` 保存上一次序列化值，仅在 `JSON.stringify` 结果变化时才写入

## 五、代码质量问题（中优先级 🟡）

### 5.1 无类型系统（TypeScript）

整个项目是纯 JavaScript，没有类型定义。对于一个有 19 个节点、30+ 个数据字段的复杂系统来说，这会导致：

- 重构时极易遗漏字段
- 组件间数据传递没有契约
- IDE 无法提供智能提示（尤其是深层嵌套的 project 对象）

`package.json` 中已经安装了 `@types/react` 和 `@types/react-dom`，说明有向 TS 迁移的基础。

**修复建议**：
- 第一阶段（低成本）：添加 JSDoc 类型注解，配合 VS Code 即可获得类型检查
  ```js
  /** @type {{ chapters: Chapter[], characters: Character[], ... }} */
  ```
- 第二阶段：将 `lib/` 下核心模块迁移为 `.ts`
- 第三阶段：组件迁移（优先迁移 store.jsx 和 lib/）

---

### 5.2 无单元测试

项目中没有任何测试文件。对于以下关键模块，缺少测试是非常危险的：
- `foreshadow-engine.js`：伏笔状态机，状态转换规则复杂
- `constraints-engine.js`：约束校验逻辑
- `progress-aggregator.js`：多层聚合计算
- `schema-validator.js`：JSON Schema 校验器
- `node-executor.js`：六步循环

**修复建议**：
- 安装 Vitest（与 Vite 生态无缝集成）
- 优先为核心 lib 模块编写单元测试
- 为 `foreshadow-engine.js` 的状态转换编写参数化测试

---

### 5.3 大量重复的 CRUD 模式

在 store.jsx 的 reducer 中，以下模式重复了 **8 次以上**：

```js
case 'ADD_XXX':     // 角色、地点、魔法、时间线、约束、伏笔、分册、章节...
case 'UPDATE_XXX':
case 'DELETE_XXX':
```

这些 case 的逻辑结构几乎完全一致，只是操作的 key 不同。

**修复建议**：使用高阶函数生成 reducer case：

```js
function createEntityCases(entityKey) {
  return {
    [`ADD_${entityKey}`]: (state, action) =>
      finalize({ ...state, [entityKey]: [...state[entityKey], action.payload] }),
    [`UPDATE_${entityKey}`]: (state, { payload: { index, patch } }) =>
      finalize({ ...state, [entityKey]: state[entityKey].map((item, i) =>
        i === index ? { ...item, ...patch } : item
      )}),
    [`DELETE_${entityKey}`]: (state, action) =>
      finalize({ ...state, [entityKey]: state[entityKey].filter((_, i) => i !== action.payload) }),
  }
}
```

### 5.5 大量 useCallback 未正确设置依赖

**文件**：[store.jsx#L703-L730](file:///d:/Documents/Desktop/Novel/src/store.jsx#L703-L730)

```js
const updateSceneContent = useCallback((content) => {
  if (!chapter) return
  dispatch({ type: 'UPDATE_SCENE_CONTENT', payload: { chapterId: chapter.id, sceneIndex: sceneIdx, content } })
}, [chapter, sceneIdx])
```

这些 useCallback 依赖于 `project` 内部派生值，但由于 `project` 是 useReducer 返回的引用（每次 dispatch 后引用会变），实际会频繁重建。**useCallback 在此场景下几乎没有优化效果**。

**修复建议**：
- 使用 `useRef` 保存 dispatch 函数（dispatch 本身稳定不变）
- 通过 action 传递动态值，而不是通过闭包捕获
